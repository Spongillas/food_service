# Food Supplier Finder — сервис поиска поставщиков продуктов питания

Прототип сервиса, который помогает быстро находить и сравнивать поставщиков
продуктов питания (ингредиенты, готовая продукция, упаковка и т.п.) по
категории товара и региону. Сделано как тестовое задание.

## Как это работает (коротко)

1. Пользователь вводит категорию товара, город/регион и (опционально)
   дополнительные пожелания — например, «нужен ХАССП» или «опт от 50 кг».
2. Бэкенд формирует промпт и отправляет его в **OpenRouter** на модель с
   суффиксом `:online` (по умолчанию `openai/gpt-4o-mini:online`).
   Суффикс `:online` включает встроенный в OpenRouter плагин веб-поиска
   (на базе Exa) — модель **сама гуглит**, открывает найденные страницы
   (сайты компаний, b2b-каталоги, маркетплейсы, справочники) и уже с их
   учётом формирует ответ. Отдельный ключ для поисковика не нужен — всё
   работает через один OPENROUTER_API_KEY.
3. Модель обязана вернуть строго структурированный JSON-массив поставщиков
   с полями: название, категория, регион, описание, сайт, ссылка-источник,
   контакты, минимальный объём заказа, ориентировочная цена, сертификаты,
   условия доставки, заметки, «уверенность» (0–1) — сколько бы данных ни
   нашлось, отсутствующие поля модель обязана оставлять `null`, а не
   выдумывать.
4. Бэкенд валидирует JSON через Pydantic, считает для каждого поставщика
   **скор рекомендации** (`confidence * 0.4 + completeness * 0.6`, где
   `completeness` — доля заполненных «полезных» полей: сайт, контакты, MOQ,
   цена, сертификаты, условия доставки) и помечает лучшего поставщика
   бейджем «Рекомендуем».
5. Результаты кэшируются в памяти на 1 час по ключу
   (категория, регион, пожелания), чтобы не дёргать AI повторно на
   одинаковые запросы.
6. Фронтенд — простой HTML/JS без сборки: форма поиска, карточки
   поставщиков, сортировка по рекомендации / цене / надёжности данных,
   ссылки на сайт и источник для проверки информации.

Если `OPENROUTER_API_KEY` не задан (или AI вернул невалидный ответ),
сервис не падает, а показывает демо-данные с явным предупреждением —
так интерфейс можно посмотреть и без ключа.

## Архитектура

```
food_service/
├── app/
│   ├── main.py                  # FastAPI-приложение, отдаёт статику и API
│   ├── config.py                # настройки из .env (pydantic-settings)
│   ├── models.py                # Pydantic-схемы запроса/ответа/поставщика
│   ├── routers/search.py        # POST /api/search, GET /api/health
│   └── services/
│       ├── openrouter_client.py # запрос к OpenRouter (LLM + веб-поиск)
│       ├── supplier_search.py   # промпт, парсинг JSON, скоринг, демо-фолбэк
│       └── cache.py             # TTL-кэш результатов поиска
├── static/                      # index.html, style.css, app.js
├── tests/test_api.py            # smoke-тесты (health, демо-поиск, валидация)
├── requirements.txt
└── .env.example
```

## Запуск локально

```bash
python -m venv .venv
.venv\Scripts\activate        # Windows
# source .venv/bin/activate   # macOS/Linux

pip install -r requirements.txt
copy .env.example .env        # или cp .env.example .env
# впишите в .env свой OPENROUTER_API_KEY (https://openrouter.ai/keys)

uvicorn app.main:app --reload
```

Открыть http://localhost:8000

Без ключа приложение тоже запустится и будет работать в демо-режиме.

## Ограничения прототипа и что можно добавить дальше

- Веб-поиск через OpenRouter `:online` — самый быстрый способ получить
  «ИИ гуглит и парсит» без отдельной инфраструктуры, но он менее
  контролируем, чем связка «свой поисковый API (Google CSE/Tavily/Serper) +
  собственный парсер страниц + LLM-извлечение». Второй вариант дал бы больше
  источников за один запрос и более стабильную повторяемость.
- Нет постоянного хранилища: результаты не сохраняются между перезапусками
  и не обогащаются вручную (заметки пользователя, статус «уже связались»).
  Логичный следующий шаг — SQLite/Postgres с таблицей поставщиков и историей
  запросов.
- Нет дедупликации поставщиков между разными запросами и разными пользователями.
- Нет аутентификации/мультитенантности — для внутреннего инструмента отдела
  закупок это можно добавить поверх.
- Сортировка по цене — эвристическая (первое число в свободном тексте),
  так как цена приходит в произвольном формате; для точной сортировки
  потребовалась бы нормализация цены в промпте (валюта + единица измерения).
